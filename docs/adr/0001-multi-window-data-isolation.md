# ADR-0001: 非主窗口数据隔离——仅使用专用 store，其余数据由主窗口事件传递

**状态**: 已采纳  
**日期**: 2026-07-22  
**决策者**: @buren_Lee

---

## 背景

应用有三个 WebView 窗口：主窗口、迷你播放器窗口（`desktop-mini`）、桌面歌词窗口（`desktop-lyric`）。实现通过主窗口传递数据子窗口接收进行通信，子窗口可创建自用store。

## 决策

非主窗口**只能使用各自的专用 store**，其余所有数据由主窗口通过 Tauri 事件系统显式传递。

具体规则：
- `desktop-mini` 只允许使用 `desktop-mini` store
- `desktop-lyric` 只允许使用 `desktop-lyric` store
- 播放状态、播放列表、歌词数据、用户信息等全部由主窗口通过事件推送
- 非主窗口的用户操作通过事件回传主窗口，由主窗口执行实际业务逻辑
- 禁止非主窗口直接引用 `music`、`list`、`lyric`、`user`、`setting` store

## 考虑的替代方案

| 方案 | 优点 | 缺点 | 为什么没选 |
|------|------|------|------------|
| 共享 Pinia store（现状） | 开发简单，直接 import 即用 | 多实例竞态、同步延迟、耦合重 | 数据一致性不可靠 |
| 通过 Tauri 全局状态共享 | 单一数据源，无需事件协议 | Tauri state 在 WebView 间不共享，技术上不可行 | 技术不可行 |
| 主窗口全权推送（本方案） | 单一数据源、事件驱动、窗口无状态、易测试 | 需设计事件协议、有网络开销（本地忽略不计） | — |

## 影响

- **新增**: 每个子窗口需要事件协议定义（`DesktopMiniEmit` 等）、主窗口需要桥接 composable 负责推送
- **修改**: 子窗口组件改为事件驱动的**轻计算视图**——主窗口推送原始数据，子窗口保留渲染相关计算（如 `activedIndex` 计算）；入口文件移除 Pinia 依赖或精简为仅专用 store
- **约束**: 新增非主窗口功能时，必须先扩展事件协议，不能直接引用主 store；新增的子窗口必须遵守同样的隔离规则
- **风险**: 事件协议变更需要主窗口和子窗口同步更新；初始化时序（子窗口创建早于主窗口 store 水合完成）需要显式处理

## 后续

- 桌面歌词窗口（`desktop-lyric`）的子窗口功能已实现，包括：播放控制、歌词偏移、字号/字体/颜色设置（弹窗修改）、窗口锁定（双击解锁）、双行/翻译布局。详见对应 spec。
- 子窗口推送数据的计算原则：**通用数据由主窗口推送，子窗口保留自身渲染计算**。主窗口推送原始进度和歌词数据，子窗口（`DesktopLyric.vue`）自行计算 `activedIndex`（当前高亮行索引）和 `nextIndex`（下行索引），因为这与子窗口的渲染节奏耦合更紧密。
- **歌词偏移独立管理**：桌面歌词窗口使用 `desktopLyricStore.offsetMap` 管理自己的偏移量，不共享 `lyric-main` 的 `offsetMap`。偏移调整直接在子窗口内完成并通过 store 持久化，bridge 只传递原始进度值。
- **初始化时序**：子窗口可能早于主窗口 store 水合完成被创建。子窗口先渲染空白/骨架状态（显示歌名兜底），主窗口就绪后通过 `Init` 事件全量推送首帧数据。
- **事件一致性**：不要求原子性。不同事件（Audio/Lyric/Progress/Fonts）独立推送，允许短暂的不一致窗口（如切歌时新 Audio 先到、旧 Lyric 尚未刷新）。依赖 throttle（33~100ms）保证旧数据在可接受时间内被覆盖，不引入事务或批量推送机制。
- **共享数据由主窗口推送**：子窗口需要的共享数据（如可用字体列表）由主窗口按需触发检测并推送，子窗口不在本地重复检测。
- **窗口锁定交互**：锁定后 `setIgnoreCursorEvents(true)` 使窗口鼠标穿透，双击拖拽区域恢复交互。
- 每个子窗口的事件协议应记录在对应的 spec 或 ADR 中，作为接口契约
